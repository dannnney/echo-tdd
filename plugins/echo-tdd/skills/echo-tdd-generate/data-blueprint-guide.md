# 测试数据蓝图设计指导

本文档指导如何基于用例清单生成测试数据蓝图——一份描述"需要什么数据、怎么创建、怎么管理"的规划文档。

---

## 1. 分层数据策略

不同测试层的数据有不同的生命周期需求。

### 层级与数据模式

| 测试层 | 数据模式 | 创建时机 | 清理时机 | 理由 |
|--------|---------|---------|---------|------|
| 认证层 | 无预置数据 / 仅凭证配置 | — | — | 只验证凭证有效性 |
| 只读操作层 | **共享 fixture** | beforeAll（全层创建一次） | afterAll（全层结束清理） | 只读操作不修改数据，可以安全共享 |
| 写操作层 | **独立数据** | beforeEach（每个用例独立创建） | afterEach（每个用例独立清理） | 写操作会修改状态，必须隔离 |
| 端到端层 | **完整生命周期** | 每个用例内部完整管理 | 用例内部清理 | 端到端故事需要从零开始 |

### 共享 fixture 的设计原则

- 只读层的 fixture 是**静态的**——一旦创建就不应被修改
- fixture 的内容应**覆盖只读层所有用例的数据需求**
- 如果某个用例需要修改 fixture 中的数据（如测试 cat 大文件），考虑将该用例移到写操作层

### 独立数据的设计原则

- 每个用例只创建**自己需要的最小数据**
- 创建的数据必须包含**可清理的引用**（如返回的 ID/token）
- 清理必须**幂等**（重复调用不报错）

## 2. 数据隔离

### 命名前缀

所有测试数据使用统一的、可识别的前缀：

```
_fz_test_       — fz 项目的测试数据前缀
_test_          — 通用前缀
test-           — 用于用户名/ID 等
```

前缀的作用：
- 避免误操作真实数据
- 清理时可以按前缀批量清理
- 快速区分测试数据和真实数据

### 命名空间

如果环境支持，使用独立的命名空间隔离测试：

```
数据库：独立的 test 数据库或 schema
文件系统：独立的测试目录
云服务：独立的测试空间/项目/bucket
```

### 时间戳唯一性

测试数据名称中包含时间戳，避免并行执行时的冲突：

```
_fz_test_{timestamp}_{name}
例：_fz_test_1711234567_root
```

## 3. 数据需求汇总规范

从用例清单中提取数据需求，汇总为蓝图。

### 提取方法

对每个用例的"前置条件"列：
1. 识别需要的**数据实体**（文件夹、文档、用户、订单等）
2. 识别数据实体的**属性要求**（名称、类型、内容、关系）
3. 标注数据的**创建通道**（通过哪个实际可用的通道创建）
4. 判断数据的**复用性**（是否可以被其他用例共享）

### 蓝图表格式

```
| 数据名称 | 类型 | 详情 | 创建方式 | 生命周期 | 被引用用例 | 优先级 |
|---------|------|------|---------|---------|-----------|-------|
```

字段说明：
- **数据名称**：唯一标识，用于引用
- **类型**：数据的类型（文件夹、文档、用户、记录等）
- **详情**：具体的属性值（名称、内容、数量等）
- **创建方式**：通过哪个通道创建（SDK/API/DB/UI），必须是阶段二确认可用的通道
- **生命周期**：何时创建、何时清理（beforeAll/beforeEach/用例内部）
- **被引用用例**：哪些用例依赖这条数据
- **优先级**：P0/P1/P2，跟随引用它的用例中最高的优先级

### 去重与合并

- 如果多个用例需要同一个数据实体（且属性要求兼容），合并为一条
- 如果属性要求冲突，分成多条独立数据
- 只读层的数据尽量合并（共享 fixture）；写操作层的数据不合并

## 4. 数据与可观测性通道的适配

数据创建方式必须适配用户实际可用的通道。

### 通道优先级

```
SDK 可用         → 优先用 SDK 创建（最灵活、最可控）
DB CLI 可用      → 直接 SQL 插入（最快、最底层）
API 可用         → 通过 API 调用创建（有业务校验）
UI 可用          → 通过浏览器操作创建（最慢、最脆弱，但万能）
```

### 通道降级

如果首选通道不可用，按优先级降级：

```
SDK 不可用 → 改用 API
API 不可用 → 改用 DB CLI
DB CLI 不可用 → 改用 UI
全部不可用 → 标注为"需要手动准备"
```

### 数据创建与清理通道可以不同

创建和清理可能走不同通道：

```
创建：通过 UI 操作创建（因为要模拟真实用户流程）
清理：通过 API 批量删除（因为 UI 清理太慢）
```

### 跨通道验证

数据创建后，通过**另一个通道验证**数据确实存在：

```
SDK 创建 → API 查询确认存在（避免 SDK 创建失败但没报错）
DB 插入 → API 查询确认可读取（避免数据在 DB 但应用层看不到）
```

## 5. 丰富完善 + 分级执行原则

数据不刻意精简，尽可能覆盖各种变体——通过优先级分级控制执行节奏。

### 覆盖要求

- 每个等价类**至少一条数据**
- 每个边界值**至少一条数据**
- 真实采样发现的变体**都要覆盖**
- 不同类型/格式**各至少一条数据**

### 优先级分级

| 级别 | 数据特征 | 执行时机 |
|------|---------|---------|
| **P0** | 核心正常路径所需的基本数据 | 每次测试执行都创建 |
| **P1** | 边界值、重要异常场景、采样发现的真实变体 | 回归测试时创建 |
| **P2** | 罕见边界、极端数据量（大文件、超长名称）、兼容性变体 | 完整验收时创建 |

### 执行模式

```
@smoke  → 只创建 P0 数据 → 运行 P0 用例 → 几分钟
@regression → 创建 P0+P1 数据 → 运行 P0+P1 用例 → 十分钟
@full → 创建全部数据 → 运行全部用例 → 可能更长
```

## 6. 数据蓝图与脚手架的衔接

数据蓝图中引用的创建/清理操作，应与阶段二脚手架中的 data-factory 和 data-cleanup 对应。

### 衔接点

| 蓝图中的描述 | 脚手架中的实现 |
|-------------|--------------|
| "SDK 创建文件夹" | `data-factory.createTestFolder()` |
| "SDK 创建 Wiki 节点" | `data-factory.createTestWikiNode()` |
| "清理所有测试资源" | `data-cleanup.cleanupAll()` |
| "SDK 查询验证" | `feishu-client.drive.exists()` |

### 新数据需求

如果蓝图中出现脚手架尚不支持的数据操作（如创建 docx 文档、上传文件），在蓝图中标注为"需要扩展 data-factory"，作为后续代码化阶段的输入。
