# 示例策略提案：后端 API（无数据库直接访问权限）

## 维度快照

| 维度 | 值 |
|------|-----|
| 代码仓库 | Express.js 后端 API 服务（TypeScript） |
| 服务端 | 本地可运行，连接远程 dev 数据库 |
| 数据库 | 远程 MySQL（dev 环境），无 CLI 访问权限 |
| 前端 | 无前端（纯 API 服务） |
| 验证通道 | API（curl/fetch）|
| 认证 | 有注册端点，可以自注册 |
| 数据策略 | 只能通过 API 创建和查询数据 |

## 测试理念

### 整体策略

这是一个典型的"黑盒 API 测试"场景——你���代码权限但无法直接访问数据库。所有数据操作和验证只能通过 API 自身完成。

我们把 API 当作一个黑盒：POST 创建数据，GET 验证数据，DELETE 清理数据。这意味着**验证通道和被测系统是同一个**，无法做独立通道验证。这是一个已知的局限，策略上我们通过以下方式弥补：

1. 用不同的 API 端点交叉验证（POST /orders → GET /orders/:id 验证）
2. 关注 API 返回值的完整性（不只检查 status code，还验证 response body）
3. 通过业务流程串联验证（创建 → 查询 → 更新 → 查询 → 删除 → 确认 404）

### 验证通道与组合

| 通道 | 工具 | 角色 |
|------|------|------|
| API | curl / supertest | 唯一通道——驱动 + 验证都靠它 |

### 集成测试分层

```
第 1 层：认证
  └─ POST /register 注册测试用户 → POST /login 获取 token
  └─ 验证：登录返回有效 token，后续请求不报 401

第 2 层：CRUD 操作
  └─ 用 token 调用各业务 API
  └─ 验证：POST 创建 → GET 查到 → PUT 更新 → GET 确认变更 → DELETE → GET 确认 404

第 3 层：权限和边界
  └─ 用不同角色的 token 测试权限
  └─ 验证：有权限的操作 200，无权限的操作 403
```

## 数据流闭环

| 问题 | 方案 |
|------|------|
| 数据从哪来？ | 通过 API 自身创建（无 DB 访问） |
| 数据怎么送入？ | API POST 调用 |
| 怎么验证？ | API GET 交叉查询（已知局限：非独立通道） |
| 怎么清理？ | API DELETE + 测试用户用特殊前缀标记（`test-*`） |

### 已知局限

这个场景没有独立验证通道。如果 POST /orders 和 GET /orders 都有 bug，我们可能发现不了。这在策略上是可接受的，因为：
- 两个端点同时出同一个 bug 的概率较低
- 全链路串联测试会暴露大部分数据不一致的问题
- 如果未来能获得 DB 读权限，可以补充独立验证

### 示例：订单创建流程

```
准备 → POST /register 注册测试用户 → POST /login 拿 token
执行 → POST /orders 创建订单
验证 → GET /orders/:id 返回 200 且数据匹配
     → GET /orders?user=test-xxx 列表中包含新订单
清理 → DELETE /orders/:id → DELETE /users/test-xxx
```
