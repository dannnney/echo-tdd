# 示例策略提案：微服务（只负责其中一个服务）

## 维度快照

| 维度 | 值 |
|------|-----|
| 代码仓库 | 订单服务（Go + Gin），微服务架构中的一个 |
| 服务端 | 本地可运行（订单服务），依赖服务在远程 dev 环境 |
| 数据库 | 远程 PostgreSQL（dev 环境），可读写 |
| 前端 | 无（或远程 dev 环境有） |
| 验证通道 | API + DB CLI（自有表）+ 依赖服务 API |
| 认证 | 内部服务间 token，dev 环境已配置 |
| 数据策略 | 自有表可读写 + 依赖服务数据通过其 API 操作 |

## 测试理念

### 整体策略

微服务场景的核心挑战是：**你控制自己的服务，但控制不了依赖方**。用户服务、支付服务、库存服务都��远程 dev 环境，可能不稳定，数据是共享的。

我们采用**由内向外的分层集成策略**：
1. 先隔离测试自己的服务（mock 或跳过依赖方）
2. 再联调测试与依赖服务的集成（连接远程 dev 环境）
3. 每一层都是集成测试——连真实 DB、调真实 API

对自己控制的数据（orders 表），用 DB 独立验证。对依赖方的数据，用它们的 API 查询验证。

### 验证通道与组合

| 通道 | 工具 | 角色 |
|------|------|------|
| DB CLI | psql | 独立验证——查自有的 orders 表 |
| 本地 API | curl | 驱动——调本地订单服务 API |
| 依赖服务 API | curl | 辅助验证——查远程用户/库存/支付服务状态 |

### 集成测试分层

```
第 1 层：订单服务自身逻辑
  └─ 本地启动，连远程 DB
  └─ 直接调订单 API，不依赖其他服务（或 mock 掉）
  └─ 验证：DB 查 orders 表

第 2 层：订单服务 + 依赖服务联调
  └─ 本地订单服务 → 远程用户服务验证用户 → 远程库存服务扣库存
  └─ 验证：DB 查 orders + 库存服务 API 查库存变化

第 3 层：完整下单流程
  └─ 创建订单 → 库存扣减 → 支付 → 订单状态流转
  └─ 验证：各个服务的 API 查询最终状态
```

## 数据流闭环

| 问题 | 方案 |
|------|------|
| 数据从哪来？ | 自有数据：SQL INSERT 进 orders 表；依赖方数据：用 dev 环境已有的或通过其 API 创建 |
| 数据怎么送入？ | SQL 直接写自有表 + 依赖服��� API 调用 |
| 怎么验证？ | SQL 查自有表（独立通道）+ 依赖服务 API 交叉查 |
| 怎么清理？ | `DELETE FROM orders WHERE user_id LIKE 'test-%'`（标记隔离） |

### 数据隔离

远程 dev 环境是共享的，其他人也在用。隔离策略：
- 测试数据统一使用 `test-` 前缀的 user_id
- 清理时只删除带 `test-` 前缀的数据
- 不要清空整个表

### 示例：创建订单流程

```
准备 → SQL INSERT 一个测试用户关联（如果需要）
     → 确认远程用户服务中测试用户存在（API GET）
     → 确认远程库存服务中商品有库存（API GET）
执行 → POST /api/orders 创建订单
验证 → psql 查 orders 表确认订单记录
     → 库存服务 API 查库存是否扣减
     → 订单状态是否为 pending_payment
清理 → DELETE FROM orders WHERE id = <test_order_id>
```
