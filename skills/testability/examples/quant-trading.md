# 示例策略提案：量化交易系统

## 维度快照

| 维度 | 值 |
|------|-----|
| 代码仓库 | Python 量化交易系统（策略引擎 + 交易执行 + 风控） |
| 服务端 | 本地可运行（策略引擎），远程券商 API |
| 数据库 | 本地 PostgreSQL，可清空（持仓和交易记录） |
| 前端 | 无前端（或简单的监控 Dashboard） |
| 可观测性 | 基础通道：数据库直查 + 日志 + 文件系统（输出比对）；组合通道：券商 SDK（查询订单状态） |
| 认证 | API Key（券商 API 密钥） |
| 数据策略 | 本地 DB 可清空重建 + 券商侧需要沙盒或 mock |

## 测试理念

### 整体策略

量化交易系统涉及实时数据流和真金白银，不能直接对真实市场做测试。我们采用**渐进式真实度**策略，分阶段验证：

1. **本地模拟**：用历史行情数据回放，验证策略引擎和风控逻辑的正确性
2. **券商模拟盘**（如果券商提供）：验证交易执行模块与券商 API 的集成
3. **真实最小验证**（用户授权后）：最小手数真实下单，验证系统打通

算法效果不在测试范围——算法是上线后看真实效果再复盘的事。我们只验证**系统是否按设计工��**。

### 可观测性方案

#### 基础通道

| 通道 | 工具 | 作为触发 | 作为观测 |
|------|------|---------|---------|
| 数据库直查 | psql | INSERT seed 持仓数据 | SELECT 验证持仓/交易记录 |
| 文件系统 | diff / snapshot | 放置历史行情文件 | 比对策略信号输出 vs 基准文件 |
| 日志系统 | 日志文件 | — | 查看风控拦截记录、异常行为 |

#### 组合通道

| 通道 | 说明 | 在测试中的作用 |
|------|------|--------------|
| 券商 SDK + API Key | 券商提供的交易 SDK | 模拟盘/真实盘中查询订单状态（渐进式第2-3阶段） |

#### 触发 × 观测 组合

| 测试场景 | 触发通道 | 观测通道 | 说明 |
|---------|---------|---------|------|
| 策略信号验证 | 文件系统（喂入历史行情） | 文件系统（输出 vs 基准比对） | 确定性最高的验证方式 |
| 风控验证 | 文件系统（极端行情数据） | 日志 + 数据库直查 | 风控日志 + DB 订单状态 |
| 模拟盘全链路 | 策略引擎（自动触发） | 数据库 + 券商 SDK | DB 持仓变化 + SDK 查订单状态 |
| 真实打通验证 | 策略引擎（最小手数） | 券商 SDK（真实订单状态） | 需用户授权，独立通道验证 |

### 集成测试分层

```
第 1 层：数据获取 + 策略引擎
  └─ 喂入历史行情 → 策略产生交易信号
  └─ 验证：信号输出 vs 预期基准文件（snapshot testing）

第 2 层：策略引擎 + 风控
  └─ 喂入极端行情数据 → 触发风控规则
  └─ 验证：风控日志 + DB 中订单状态为"��拦截"

第 3 层：全链路（模拟环境）
  └─ 历史行情 → 策略信号 → 模拟盘下单 → DB 记录
  └─ 验证：DB 持仓变化符合预期

第 4 层：真���打通验证（需用户授权）
  └─ 真实行情 → 策略信号 → 真实盘最小手数下单
  └─ 验证：券商 API 查询订单状态
```

### 渐进式真实度

| 阶段 | 环境 | 目的 | 风险 |
|------|------|------|------|
| 本地模拟 | 历史数据回放 + mock 撮合引擎 | 验证策略逻辑、风控规则 | 零 |
| 券商模拟盘 | 券商提供的 sandbox（如果有） | 验证交易执行、API 集成 | 零 |
| 自建 Mock（备选） | 本地 mock 券商 API | 当券商无模拟盘时使用 | 零 |
| 真实最小验证 | 真实券商账户，最小手数 | 验证系统真实打通 | 低（需用户授权） |

## 数据流闭环

| 问题 | 方�� |
|------|------|
| 数据从哪来？ | 历史行情文件（CSV/Parquet）作为确定性输入 |
| 数据怎么送入？ | mock 数据源按时间戳推送 + DB seed 初始持仓 |
| 怎么验证？ | DB 查持仓/交易记录 + 信号输出 vs 基准文件比对 |
| 怎么清理？ | DB reset（本地可清空） |

### 示例：策略引擎 + 风控验证

```
准备 → DB reset + seed 初始持仓（1000股AAPL + 现金100000）
     → 准备两份历史行情：正常行情 + 触发风控的极端行情
执行 → 用正常行情跑策略引擎，收集交易信号
     → 用极端行情跑策略引擎，触发风控
验证 → 正常行情：信号输出 vs 基准文件一致
     → 极端行情：风控日志中有拦截记录 + DB 中订单标记为 blocked
     → DB 最终持仓 vs 预期持仓一致
清理 → DB reset
```
