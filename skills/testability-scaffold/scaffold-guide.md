# 脚手架生成指导原则

本文档提供通用的脚手架生成原则。不预设特定语言/框架的代码模板——Agent 应根据项目实际使用的技术栈来决定具体实现。

---

## 1. 技术栈决策

### 判断依据

从策略文档第 2 节"环境画像"中提取技术栈信息，然后：

| 需要决定 | 判断方法 |
|---------|---------|
| 编程语言 | 跟随项目主语言 |
| 测试框架 | 项目已有 → 用已有的；未配置 → 根据下表推荐 |
| 模块系统 | 项目用 ESM → 用 ESM；用 CJS → 用 CJS |
| 代码风格 | 看项目的缩进、命名、格式化配置 |
| 包管理器 | 看 lock 文件：package-lock.json → npm，yarn.lock → yarn，pnpm-lock.yaml → pnpm |

### 测试框架推荐（仅当项目无现有配置时）

| 技术栈 | 推荐测试框架 | 理由 |
|--------|-------------|------|
| Node.js / TypeScript | Vitest | 原生 TS 支持、与 Vite 生态兼容、快速 |
| Node.js (无 TS) | Vitest 或 Jest | 都是主流选择 |
| Python | pytest | 事实标准 |
| Go | 标准库 testing | Go 的惯例 |
| 前端 E2E | Playwright | 跨浏览器、稳定、API 直观 |

**原则**：有现有配置就用现有的，没有才推荐。不要引入用户不熟悉的工具。

---

## 2. 脚手架结构

### 必须包含的 5 个模块

无论什么技术栈，脚手架都必须包含这 5 个模块。具体文件名和实现方式跟随技术栈。

#### 模块 1：认证 Helper

**职责**：封装策略文档第 5 节的认证方案为可复用的函数。

**必须提供的能力**：
- 获取认证凭证（token/cookie/session）
- 缓存凭证，避免每个测试都重新获取
- 刷新过期凭证（如果适用）
- 验证凭证有效性的检查函数

**设计原则**：
- 凭证信息从环境变量读取，不硬编码
- 提供工厂函数，支持创建不同角色的认证（如果策略涉及多角色）
- 凭证对象可以传给通道客户端使用

#### 模块 2：通道客户端

**职责**：封装每个已确认可用的观测通道为客户端实例。

**必须提供的能力**：
- 每个可用通道一个客户端工厂函数
- 客户端自动携带认证信息
- 提供通用的查询/验证方法

**设计原则**：
- 基础通道和组合通道都要封装
- 不可用的通道：定义接口但标注 `TODO`，便于未来扩展
- 客户端应该是无状态的（每次调用独立）

**示例接口模式**（伪代码）：
```
// SDK 组合通道
createSDKClient(auth) → client
  client.query(params) → result
  client.create(data) → resource
  client.delete(id) → void

// DB 基础通道
createDBClient(connectionString) → client
  client.query(sql) → rows
  client.close() → void

// API 基础通道
createAPIClient(baseURL, auth) → client
  client.get(path) → response
  client.post(path, data) → response
  client.delete(path) → response
```

#### 模块 3：数据工厂

**职责**：封装策略文档第 4 节的数据准备方案。

**必须提供的能力**：
- 创建各类测试数据的函数
- 返回创建的资源 ID/引用（供后续验证和清理使用）
- 数据隔离（统一前缀/命名空间）

**设计原则**：
- 使用策略文档中确认的数据送入通道（DB 直写 / API 调用 / SDK 调用 / UI 操作）
- 所有测试数据使用可识别的前缀（如 `_test_`、`test-`），避免污染真实数据
- 工厂函数返回的资源引用应足够清理使用
- 支持批量创建和单个创建

#### 模块 4：数据清理

**职责**：封装策略文档第 4 节的清理方案。

**必须提供的能力**：
- 按资源 ID 清理单个资源
- 按前缀/命名空间批量清理
- 全局清理函数（清理所有测试数据）

**设计原则**：
- 清理函数应该是幂等的（重复调用不报错）
- 清理失败不应阻塞测试流程（打 warning 但不 throw）
- 注册为测试框架的 afterAll/teardown 钩子

#### 模块 5：Smoke Test

**职责**：最小可运行测试，验证整个脚手架链路。

**必须覆盖的链路**：
```
认证 → 获取凭证成功
    → 数据工厂 → 创建一个最小测试资源
        → 通道客户端 → 通过观测通道验证资源存在
            → 数据清理 → 清理测试资源
                → 通道客户端 → 验证资源已不存在
```

**设计原则**：
- 一个测试文件，一个测试用例
- 测试应在 30 秒内完成
- 测试结束后环境应恢复原状（无残留数据）

## 3. 环境变量管理

### `.env.test` 模板

脚手架必须生成一个 `.env.test` 模板文件，列出所有需要的环境变量：

```
# 认证相关
APP_ID=your_app_id_here
APP_SECRET=your_app_secret_here

# 数据库相关（如果适用）
DB_HOST=localhost
DB_PORT=5432
DB_USER=test
DB_PASSWORD=test
DB_NAME=test

# 其他通道相关
API_BASE_URL=http://localhost:3000
```

**原则**：
- 提供有意义的占位值（不是空字符串）
- 敏感值用 `your_xxx_here` 提示用户填写
- 本地开发的默认值可以直接填写（如 `localhost:5432`）
- 加入 `.gitignore`

### 环境变量加载

- Node.js：使用 `dotenv` 或测试框架内置的 env 加载
- Python：使用 `python-dotenv`
- Go：使用 `godotenv` 或 os.Getenv

## 4. 测试配置

### 全局 Setup/Teardown

```
setup:
  1. 加载 .env.test
  2. 初始化认证（获取 token）
  3. 初始化通道客户端
  4. （可选）创建测试命名空间/目录

teardown:
  1. 全局清理测试数据
  2. 关闭通道客户端连接
  3. 输出清理报告
```

### 超时配置

集成测试涉及真实网络请求，默认超时应该比单元测试长：
- 单个测试：30 秒
- 全局：5 分钟

---

## 5. 代码质量要求

- 生成的代码必须**可直接运行** — 不是伪代码
- 所有函数有清晰的入参和返回值类型（如果是 TS/Go 等有类型系统的语言）
- 错误处理：网络请求、DB 操作等都要有合理的错误处理和重试
- 日志：关键操作打 debug 级别日志，方便排查
- 注释：只在不明显的逻辑处加注释，不要过度注释
