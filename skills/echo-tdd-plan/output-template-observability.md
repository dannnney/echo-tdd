## 基础通道

基础通道是可以独立使用的观测和触发工具，是构建测试的基础单元。

| 通道 | 工具 | 作为触发 | 作为观测 | 角色（主要/辅助） | 可用性 | 说明 |
|------|------|---------|---------|-----------------|-------|------|
| [通道名] | [具体工具] | [可触发的操作] | [可观测的结果] | [主要观测/辅助验证] | [已确认/待验证/不可用] | [补充说明] |

**示例**：
| 通道 | 工具 | 作为触发 | 作为观测 | 角色 | 可用性 | 说明 |
|------|------|---------|---------|------|-------|------|
| 浏览器 | Playwright | 用户交互（点击、输入） | DOM 状态、Console、Network、Visual | 主要观测 | 已确认 | 前端应用的核心观测平台 |
| 数据库 CLI | psql | 直接写入测试数据 | SQL 查询验证数据 | 辅助验证 | 已确认 | 独立验证数据层，避免被测 API 自证 |
| HTTP API | curl / axios | 调用 REST 端点 | 响应状态码、body | 主要观测 | 已确认 | 后端 API 的主要验证方式 |
| 标准输出 | - | - | stdout/stderr/exitCode | 主要观测 | 已确认 | CLI 工具的主要观测通道 |

### 浏览器子通道展开（如适用于 WebUI 项目）

对于 WebUI/前端项目，浏览器不仅是触发器，更是最核心的观测通道。必须将浏览器子通道展开说明：

| 子通道 | 观测内容 | 典型用途 | 工具/API |
|--------|---------|---------|---------|
| **DOM 状态** | 元素存在性、文本内容、属性值、CSS 类 | 验证 UI 渲染结果、状态变化 | `page.locator()`, `expect(element).toBeVisible()` |
| **Console** | 日志输出、警告、错误 | 验证前端逻辑执行、捕获异常 | `page.on('console')`, `page.evaluate()` |
| **Network** | HTTP 请求/响应、状态码、请求体/响应体 | 验证前后端交互、API 调用 | `page.on('request')`, `page.on('response')` |
| **Visual** | 截图、视觉回归 | 验证 UI 样式、布局正确性 | `page.screenshot()`, visual diff tools |
| **Storage** | localStorage、sessionStorage、Cookie | 验证状态持久化、登录态 | `page.evaluate(() => localStorage.getItem())` |
| **URL/路由** | 当前 URL、路由变化 | 验证导航、跳转逻辑 | `page.url()`, `expect(page).toHaveURL()` |

**关键原则**：对于前端项目，不要把浏览器仅当"触发器"——它同时是最核心的观测通道。DOM、Console、Network 等子通道能提供丰富的验证维度。

## 触发 × 观测 矩阵（完整版）

每个功能模块的完整触发和观测策略。这是测试用例设计的基础——每个用例的触发方式和验收方式必须从这个矩阵中选择。

| 功能模块 | 触发通道 | 主要观测通道 | 辅助验证通道 | 说明 | 数据准备 | 数据清理 |
|---------|---------|------------|------------|------|---------|---------|
| [模块名] | [如何触发] | [主要观测什么] | [辅助验证什么] | [为什么这样组合] | [如何准备数据] | [如何清理] |

**示例（Next.js 全栈应用）**：
| 功能模块 | 触发通道 | 主要观测通道 | 辅助验证通道 | 说明 | 数据准备 | 数据清理 |
|---------|---------|------------|------------|------|---------|---------|
| 用户注册 | 浏览器填写表单 | DOM 状态（成功提示）+ URL 跳转 | DB 查询 `users` 表 | UI 操作 + 独立数据验证，避免只检查前端反馈 | 无需准备 | DB reset |
| 用户登录 | 浏览器输入凭证 | DOM 状态（用户信息显示）+ Cookie | DB 查询 `sessions` 表 | 验证登录态在前后端都生效 | DB seed 创建测试用户 | DB reset |
| 创建文档 | API POST `/api/docs` | API 响应（201 + doc ID） | DB 查询 `documents` 表 | API 触发 + 数据层确认，测试后端逻辑 | DB seed 创建认证用户 | API DELETE 或 DB reset |
| 文档列表 | 浏览器打开列表页 | DOM 检查（文档卡片） | DB 查询验证数量一致 | 前端渲染 + 数据源确认 | DB seed 创建 5 个测试文档 | DB reset |
| 编辑文档 | 浏览器编辑器输入 | Network 请求（PATCH 成功） + DOM 更新 | DB 查询 `documents.content` | 验证前端编辑 → API 保存 → DB 持久化 | DB seed 创建文档 | DB reset |
| 删除文档 | 浏览器点击删除按钮 | DOM 状态（文档消失） | DB 查询（soft delete 或物理删除） | UI 操作 + 数据层确认删除生效 | DB seed 创建文档 | DB reset（已删除） |
| 分享文档 | API POST `/api/docs/:id/share` | API 响应（分享链接） | 浏览器访问链接验证可访问 | 后端生成 + 前端验证，跨通道验证 | DB seed 创建文档 | DB 删除分享记录 |
| WebSocket 实时协作 | 浏览器 1 编辑 + 浏览器 2 观察 | 浏览器 2 的 DOM 更新 | Network 监听 WS 消息 | 双浏览器实时同步验证 | DB seed 创建文档 | 关闭 WS 连接 + DB reset |

**示例（CLI 工具）**：
| 功能模块 | 触发通道 | 主要观测通道 | 辅助验证通道 | 说明 | 数据准备 | 数据清理 |
|---------|---------|------------|------------|------|---------|---------|
| 初始化项目 | CLI `app init` | stdout（成功提示）+ exitCode 0 | 文件系统检查（配置文件存在） | CLI 输出 + 文件验证 | 临时目录 | 删除临时目录 |
| 同步数据 | CLI `app sync` | stdout（进度条）+ exitCode 0 | SDK/API 查询验证同步结果 | CLI 反馈 + 独立验证同步生效 | SDK 创建测试数据 | SDK 删除测试数据 |
| 配置修改 | CLI `app config set key=value` | stdout（配置已更新） | 读取配置文件内容 | CLI 反馈 + 文件验证 | 初始化默认配置 | 恢复默认配置 |

**矩阵设计原则**：
1. **主要观测是核心**：必须能直接反映功能是否成功
2. **辅助验证是保障**：通过独立通道确认，避免被测模块自证
3. **触发方式匹配实际使用**：用户怎么用就怎么触发
4. **数据准备和清理配套**：确保测试可重复、环境干净

## 约束与局限

### 通道约束

[详细说明哪些通道不可用、为什么、有什么影响]

**格式**：
- **[通道名] 不可用**
  - 原因：[为什么不可用，如"无 DB 访问权限"/"第三方服务无沙盒环境"]
  - 影响：[哪些测试场景受影响]
  - 应对：[降级方案是什么]

**示例**：
- **数据库 CLI 不可用**
  - 原因：远程数据库仅提供 API 访问，无直接 SQL 权限
  - 影响：无法通过 DB 查询做独立验证，无法直接 seed 测试数据
  - 应对：
    - 数据准备：改用 API 创建测试数据（较慢但可行）
    - 数据验证：改用 API 查询验证（降级为非独立验证，但可接受）
    - 数据清理：改用 API DELETE（需确认删除权限）

- **飞书 webhook 回调不可用**
  - 原因：本地环境无公网 IP，飞书无法回调
  - 影响：无法测试 webhook 触发的功能（如文档变更通知）
  - 应对：
    - 开发环境：mock webhook 回调（手动触发回调逻辑）
    - 集成测试：使用 ngrok 暴露本地服务（需用户安装）
    - 最终验证：在有公网 IP 的测试环境部署验证

### 功能约束

[详细说明哪些功能/模块无法独立验证、为什么、有什么替代方案]

**格式**：
- **[功能/模块] 无法独立验证**
  - 原因：[为什么无法独立验证，如"只有前端展示"/"无独立观测通道"]
  - 影响：[验证可靠性降低到什么程度]
  - 应对：[如何尽可能增强验证可靠性]

**示例**：
- **前端样式和布局无法独立验证**
  - 原因：只有浏览器渲染结果，无独立的样式数据源
  - 影响：只能通过截图或 DOM 结构验证，无法确认视觉一致性
  - 应对：
    - 使用 visual regression testing（截图对比）
    - 关键布局用 DOM 结构 + CSS 类验证
    - 手工验证关键页面的视觉效果

- **第三方 OAuth 登录流程无法完全自动化**
  - 原因：第三方登录页面不在我们控制范围
  - 影响：无法端到端自动化测试 OAuth 流程
  - 应对：
    - Mock OAuth 回调（开发环境）
    - 手工登录后导入 token（集成测试）
    - 在测试环境用测试账号手工验证（上线前）

### 降级方案总结

当理想的观测通道不可用时，按以下优先级降级：

```
理想（独立验证）
  ↓ 不可用
次优（交叉验证）
  ↓ 不可用
可接受（自身验证 + 多维度检查）
  ↓ 不可用
最低（手工验证）
```

**具体示例**：
- 数据验证：DB 直查（理想）→ SDK 查询（次优）→ API 查询（可接受）→ UI 显示检查（最低）
- 数据准备：DB seed（理想）→ SDK 创建（次优）→ API 创建（可接受）→ UI 操作创建（最低）
- 异步操作验证：轮询 DB 查询（理想）→ 轮询 API 查询（次优）→ 固定等待 + UI 检查（最低）

**原则**：每个降级都要明确说明为什么降级、影响是什么、如何尽可能保证可靠性。

## 渐进式真实度（如适用）

当系统涉及**金钱、时间流、外部副作用**（支付、交易、短信、邮件、定时任务等）时，采用渐进式验证策略。

| 阶段 | 环境 | 适用场景 | 验证目标 | 配置要求 | 风险等级 | 说明 |
|------|------|---------|---------|---------|---------|------|
| [阶段名] | [什么环境] | [什么功能] | [验证什么] | [需要什么配置] | [风险评估] | [补充说明] |

**示例（支付系统）**：
| 阶段 | 环境 | 适用场景 | 验证目标 | 配置要求 | 风险等级 | 说明 |
|------|------|---------|---------|---------|---------|------|
| Mock 本地模拟 | 本地 mock 支付网关 | 开发阶段快速迭代 | 验证支付逻辑正确性（订单创建、状态更新、回调处理） | 本地 mock server | 零风险 | 无真实扣款，可随意测试 |
| 支付宝沙盒 | 支付宝沙盒环境 | 集成测试主力 | 验证与支付宝的集成打通（请求格式、签名、回调） | 沙盒账号、沙盒 App ID | 低风险 | 接近真实，但不扣真钱 |
| 真实 1 分钱验证 | 支付宝生产环境 | 上线前最后验证 | 验证生产环境真实打通 | 真实 App ID、用户授权 | 需用户授权 | 仅验证打通，不验证业务逻辑 |

**示例（短信发送）**：
| 阶段 | 环境 | 适用场景 | 验证目标 | 配置要求 | 风险等级 | 说明 |
|------|------|---------|---------|---------|---------|------|
| Mock 短信网关 | 本地 mock | 开发阶段 | 验证短信发送逻辑（触发时机、内容模板） | 本地 mock server | 零风险 | 不发送真实短信 |
| 短信服务商测试号 | 阿里云短信测试 | 集成测试 | 验证短信服务集成（API 调用、错误处理） | 测试 API Key、测试手机号 | 低风险 | 仅发送到测试号，不扣费 |
| 真实号码验证 | 阿里云短信生产 | 上线前验证 | 验证生产环境打通 | 生产 API Key、测试手机号授权 | 需用户授权 | 仅发送 1-2 条验证 |

**关键原则**：
1. **优先用服务商提供的沙盒** — 最省力、最接近真实、零风险
2. **没有沙盒则自己 mock** — 可控但需维护 mock 逻辑
3. **沙盒跑通后再考虑真实最小验证** — 目的是验证「系统打通」，不是验证业务逻辑
4. **不是所有系统都需要三层** — 简单项目只需 mock 即可
5. **真实验证需要用户明确授权** — 涉及真实金钱/短信/邮件时，必须提前告知用户

**不适用场景**：
- 纯内部逻辑（如算法、数据处理）— 直接真实环境测试即可
- 无外部副作用的 API 调用（如查询操作）— 直接调用真实 API
- 本地可完全控制的资源（如本地数据库、文件系统）— 直接操作真实资源

- **多浏览器并发**：Playwright 启动多个 browser context，模拟多用户

- **压力测试**（如适用）：短时间内大量请求，验证系统稳定性

### 数据隔离策略

[说明如何避免测试数据互相干扰]

**常见模式**：
- **前缀/命名空间**：所有测试数据使用统一前缀（如 `_test_`）
