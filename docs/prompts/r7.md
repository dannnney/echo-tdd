1. plan skill 在第四阶段方案确认时，还是应该走“向用户提问”的机制，现在一次性输出好几问题、每个问题好多个选项，一大段文字让用户依然很不好用
2. 我们在 @docs/echo-tdd 目录下，输出文档时，存储的路径需要增加时间，类似于 superpowers ，可以参考一下  @others/superpowers 它的源代码
3. 优化 plan 时，沟通的一些问题：
    - 我们是一个 e2e 的集成测试方案，所以不可能存在任何 mock 的可能性，也就是说我们不应该在规划 plan 时还去询问要不要考虑 mock ，这个是严重错误的
    - 在沟通 plan 时，也就是在 verify 阶段前，不要撰写或者执行任何的代码进行环境验证，这样会耽误时间和浪费 token，在 verify 阶段来做即可。就算最终 verify 失败，和用户沟通后，也可以再来修改 plan
    - 在 plan SKILL 的描述中，要重点强调 MacOS 这一类的 APP 的验证主要是要以 mac-use SKILL 为主要验证通道， mac-use 是我们在 MacOS 上首选的自动化测试方案。这个要明确出来。
    - 避免沟通测试规模的问题，我们现在是做方案阶段，方案就是要大而全，要尽全力去探索测试的极限边界，并且一切都已真实环境的为主，不考虑 Mock。当然，我们会在第三个阶段去做测试用例的分级规划，所以第一阶段只需要规划出覆盖面最广最全的范围即可，避免、减少和用户的沟通
    - 在 plan 的文档中，还应该记录一下原始需求的文档位置，方便后续 skill 跟踪 
4. 每个阶段执行完毕后，会提醒后续下一个 SKILL 的启动，这里的名字不太对，你可以扫描一下代码，现在的 SKILL 名字有所变化
5. 在 generate 阶段，增加自动测试分级的规划
6. 在 verify 阶段，可以以当前 cwd 目录为根目录，尝试推测当前目录是不是用户打算进行开发的源代码目录，这里可能有两种情况，一种是用户还没有创建任何的工程文件，一种也许已经存在工程文件了，这是一个老的项目。无论是否存在代码，如果有可能推测出来，就尽可能不要和用户沟通，自动进行后续的流程，比如创建测试、验证环境，跑冒烟等等，减少用户沟通和干预。不管是 Playwright 操作浏览器，还是 mac-use 操作 MacOS ，还是 postgres cli 操作数据库，都尽力而为的自己验证、不要停、不要问题。
