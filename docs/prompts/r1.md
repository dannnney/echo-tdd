我正在设计一个基于用户需求`动态的创建测试方案`的 skills
它的目标是希望根据用户输入的需求、场景，我们为用户量身打造一套具有高度可执行的测试方案
这个测试方案的目标是为了让我们的 Vibe Coding 系统具有一种自反馈的机制

也就是说当 Vibe Coding 生成的代码运行起来后，可以基于我们的单元测试、e2e测试、随机漫游测试等测试手段，发现问题，我们把发现的问题送入 Claude Code 等开发工具，让它继续持续的修复和运行

所以整个系统可以稳定运行的一个核心问题就是我们要梳理清楚用户的目标和 spec （目标和 spec 暂时不在这个讨论中），当我们有了 目标和 spec 后，就需要一个可以真实测试的环境，
比如访问浏览器、点击页面
比如基于 postgres cli 直接访问数据库
比如通过 curl 或者浏览器，拿到 API 请求的结果
比如提供 websocket 联通服务器后，可以等待期待的 event 等

另外一个问题就是我们要考虑到开发者的真实环境，他可能是个人开发者，团队开发者，也有可能是超级大公司内部职员。
我们有可能是前端开发、后端开发、数据库开发，也有可能是全栈开发、或者可能是APP开发者，或者是桌面端开发者。

所以他们有很多的场景是及其不一样的。

我举一个例子，如果他是一个大公司的开发者，他有可能只需要做 Server API 开发，数据库他不需要处理。因为在公司内部已经有测试环境稳定的数据库了。这种情况下，他的数据库应该会有一些 dev 权限，可以去读取数据库做验证。但是因为很多人用它不能轻易的去清空这些数据库。

如果他是一个小开发者，他可能是自己本地搭建的数据库，这种情况下，我们就可以通过在做测试数据准备的时候，提前去清空数据库或者筛入一些测试数据等等。

还有一些情况，如果只是一个 server api 的开发者，他的验证边界可能只需要确保到我们通过浏览器拿到 json 输出即可。

有可能他是一个前端开发者，他的API 是别人开发的，所以他只需要做对接和验证，他主要依靠浏览器来进行驱动验证。


也有可能他是个全栈开发，前端后端数据库都是他来负责，并且数据库在本地，他的验证可以直接从浏览器来进行驱动，但是我们可以在数据库层通过 cli 来拿数据进行验证

我说这么多场景，其实就是为了提醒你不同的开发者，不同的场景，他的可验证方案是完全不一样的。

我希望设计的这个skill，你可以先从用户输入的需求来进行一次推测。
也可以从他提前给到你的代码库来进行一次推测。
如果还是不能完全决定，你就可以对用户提问，也跟他沟通，他也可以提供到的所有环境。
如果他可以完全跑通注册，那就不需要他提供登录密码这一类的，你自己可以通过基于浏览器来实现用户的注册，从而拿到登录台自我验证。

那如果做不到，比如是公司内部很大的系统，你就可以让他提供用户名和密码，你自己来进行验证。
如果这个也做不到的话，我们可以考虑让他提供cookie或者 auth 文件，我们自己来导入进行验，等等。

还有如果是 Desktop 开发者，我们可以使用 mac-use 这样的 skill ，可以控制 mac 电脑，也可以实现类似于 playwright-cli 这样的自动化测试方案。

我说这么多方案就是提醒你，你要跟用户去探讨和穷举所有的这些案例。在最终为他设计一个端到端的测试方案，确保我们是可以做到自反馈机制。
