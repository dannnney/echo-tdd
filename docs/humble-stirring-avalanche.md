# 计划：将"可观测性"概念融入 Testability Skill

## Context

当前 testability skill 的维度4"验证通道"列举了一些验证手段（浏览器、DB CLI、API 等），但缺乏系统化的理论框架。用户提出的"可观测性"概念将测试验证提升到了更本质的层面：

1. **触发和观测是两个独立动作**，可以走不同通道（浏览器触发 → psql 观测）
2. **观测通道分两层**：基础通道（10 种）+ 组合通道（SDK、自建 API、脚本编排）
3. **组合通道不是锦上添花**，在很多场景下是唯一可行的观测路径（如飞书 SDK + 秘钥 vs 裸 HTTP）

这些概念需要融入现有 skill，替换当前较粗糙的"验证通道维度"。

---

## 修改范围

### 1. `dimensions.md` — 重构维度4

**当前状态**：维度4"验证通道"只是一个简单列表（浏览器、DB CLI、API、WebSocket、桌面自动化、移动端、日志、文件系统）

**改为**：维度4 重命名为"可观测性"，包含：

- **可观测性理念说明**：模拟人类测试工程师基于现实条件选择观测方式
- **触发通道 vs 观测通道** 的概念区分，强调它们可以自由组合
- **基础通道**（10 种，分3大类）：
  - 界面观测：浏览器页面、APP/客户端界面、浏览器 DevTools
  - 数据观测：数据库直查、API 请求、缓存/队列/事件流、文件系统
  - 过程观测：命令执行结果、进程流式输出、日志系统
- **组合通道**（基于基础通道封装，提供质变的观测能力）：
  - SDK + 凭证（如飞书 SDK、AWS SDK）
  - 自建辅助 API（临时开发用于测试验证的端点）
  - 自动化脚本（编排多个基础通道）
- **组合通道的推断线索**：项目中的 SDK 依赖、已有的辅助脚本等
- **触发×观测组合示例**（从 r4.md 中的例子整合）
- 更新维度依赖关系图

### 2. `SKILL.md` — 更新核心流程

- Phase 1（代码库扫描）：增加对 SDK 依赖和现有辅助工具的扫描指导
- Phase 3（维度补全）：可观测性维度的提问策略——先识别基础通道，再探测组合通道
- 核心原则中补充"可观测性优先"相关原则

### 3. `output-template.md` — 更新输出模板

- "验证通道"部分重构为"可观测性方案"
- 增加"触发通道 → 观测通道"的组合表
- 增加组合通道说明区域

### 4. `examples/` — 更新示例文件

对现有 6 个示例，用新的可观测性术语更新其中的验证通道部分。重点更新：
- `cli-remote-api.md` — 已经是 SDK 观测的典型案例，用新术语重新表述
- `api-no-db-access.md` — 无 DB 访问时的观测策略，补充组合通道思路
- 其余 4 个文件做术语对齐
