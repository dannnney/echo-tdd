> 生成时间：2026-04-05
> 项目：Echo-TDD
> 需求来源：[docs/prompts/r7.md](../../prompts/r7.md)
> 优化轮次：第二轮（共三轮）

---

## 1. 背景

第一轮优化已解决文档输出超时问题。本轮聚焦交互体验优化。

### 待解决问题（第二轮）

**P1 优先级**：
1. **Phase 4 方案确认交互低效**：当前一次性展示 4 个决策点的长文本，用户需要阅读大量内容后才能回复
2. **verify 阶段沟通次数过多**：多次询问用户（探测前确认、失败后询问、文件位置确认等），打断流程

### 目标

1. **Phase 4 使用结构化提问**：采用平台的结构化提问工具，让用户可以逐个决策、快速选择
2. **verify 阶段减少沟通**：开始时一次性收集所有信息，然后自动执行，只在失败时才停下来

## 2. Phase 4 结构化提问设计

### 2.1 当前问题

Phase 4 当前输出格式：
```markdown
**决策 1：可观测性重心与通道角色**
[长段落说明]
→ 确认 / 不确定，先跳过 / 需要修改：___

**决策 2：触发 × 观测的核心组合**
[长表格]
→ 确认 / 不确定，先跳过 / 需要修改：___

[... 4 个决策点]

请逐条回复，也可以直接说"全部确认"。
```

**问题**：
- 用户需要阅读大量文本
- 回复格式不统一（文本输入容易格式错误）
- 无法快速选择，体验不够流畅

### 2.2 新设计：平台无关的结构化提问

**设计原则**：
- 抽象化描述，不绑定特定平台工具
- Claude Code 使用 AskUserQuestion
- 其他平台使用各自的结构化提问工具
- 如果平台不支持，降级为传统文本提问

**实现方式**（在 SKILL.md 中描述）：

```markdown
### Phase 4 实现方式

**优先使用平台的结构化提问工具**（如果支持）：

**Claude Code**：使用 `AskUserQuestion` 工具，一次展示 4 个问题，每个问题包含：
- question：决策点的具体内容
- header：决策标签（如"观测重心"、"触发组合"）
- options：
  - 选项 1：✅ 确认 - 这个决策没问题
  - 选项 2：⏭️ 跳过 - 暂时不确定，标记为待确认
  - （自动的 Other 选项）- 用户输入具体的修改意见

**其他平台**：使用各自提供的结构化问答工具（如 Codex 的 ask_user、Cursor 的 prompt_user 等）

**降级方案**（平台不支持结构化提问时）：
逐个展示决策点，每个决策点后等待用户文本回复。格式：
```
【决策 1：可观测性重心】
[决策内容]

请回复：
- "确认" - 继续
- "跳过" - 标记为待确认
- 或直接输入修改意见
```
```

### 2.3 AskUserQuestion 使用示例


- 用户可以快速点击确认

- UI 友好，不需要记忆回复格式

## 3. verify 阶段自动化设计

### 3.1 当前问题

verify 阶段当前的沟通点：
1. **Phase 1 探测前**：每次探测前询问用户确认
2. **Phase 2 失败处理**：每个失败项询问用户选择修复方案
3. **Phase 3 脚手架生成前**：询问文件放置位置
4. **Phase 3 生成 .env.test 后**：检查 .gitignore

**问题**：
- 多次打断流程
- 用户需要多次回复
- 降低自动化程度

### 3.2 新设计：前置收集 + 自动执行

**核心思路**：在 Phase 0 和 Phase 1 之间插入一个 **Phase 0.5：前置信息收集**，一次性收集所有需要的信息，然后 Phase 1-4 自动执行。

**Phase 0.5 收集的信息**：
1. **脚手架目录位置**：默认 `test/`，用户可自定义
2. **是否需要创建 .env.test**：如果不存在，是否创建
3. **探测执行模式**：
   - 自动执行（推荐）- 只读探测自动执行，写操作探测在执行前展示清单后自动执行
   - 逐项确认 - 每项探测前都询问（适合谨慎场景）

**Phase 1-4 行为调整**：
- **Phase 1 探测**：
  - 只读探测（版本检查、连通性测试）：自动执行，不询问
  - 写操作探测（创建测试数据）：执行前展示清单，告知将要执行的操作，然后自动执行
  - 失败时：记录 FAIL，继续下一项（不停下来询问）
- **Phase 2 失败处理**：
  - 展示所有失败项
  - 自动提出降级方案（不询问用户选择）
  - 只在**关键层失败且无降级方案**时才询问用户
- **Phase 3 脚手架生成**：
  - 使用 Phase 0.5 收集的目录位置
  - 自动生成 .env.test 模板
  - 自动添加到 .gitignore（如果没有）
- **Phase 4 smoke test**：
  - 自动执行
  - 失败时展示错误，询问是否修复后重试

### 3.3 Phase 0.5 实现

**使用结构化提问工具**（Claude Code 用 AskUserQuestion）：

```javascript
{

    {

        {
          label: "test/",
          description: "默认测试目录（推荐）"

          description: "另一种常见的测试目录"
        },
        {
          label: "__tests__/",
          description: "Jest 默认目录"
        }
        // Other 选项：用户自定义路径
      ]
    },
    {
      question: "检测到 .env.test 不存在，是否需要创建？",
      header: ".env.test",
      multiSelect: false,
      options: [
        {
          label: "✅ 创建",
          description: "自动创建 .env.test 模板文件（推荐）"
        },
        {
          label: "⏭️ 跳过",
          description: "我会手动创建"
        }
      ]
    },
    {
      question: "探测执行模式？",
      header: "执行模式",
      multiSelect: false,
      options: [

```

**收集后的行为**：
- 将用户选择记录为变量（`scaffoldDir`、`createEnvTest`、`probeMode`）
- Phase 1-4 根据这些变量自动执行
- 在执行过程中不再询问用户（除非遇到无法自动处理的失败）

## 6. 风险和缓解

### 风险 1：结构化提问工具不可用

**表现**：某些平台不支持结构化提问

**缓解**：
- 提供降级方案（传统文本提问）
- 在 SKILL.md 中明确说明降级逻辑
- 保证无论哪种方式都能完成流程

### 风险 2：自动执行导致意外操作

**表现**：用户担心自动执行会创建不想要的数据

**缓解**：
- 写操作探测前展示清单
- 创建的测试数据立即清理
- 提供"逐项确认"模式供谨慎场景使用
- 所有测试数据使用明显的前缀（如 `_echo_tdd_test_`）

### 风险 3：前置收集信息不足

**表现**：后续流程中仍然需要询问用户

**缓解**：
- 仔细分析流程，确保 Phase 0.5 收集全部信息
- 如果后续发现需要新信息，记录为后续优化点
- 保留关键决策点的用户确认（如关键层失败）
