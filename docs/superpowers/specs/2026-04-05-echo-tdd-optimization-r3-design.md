### 2.3 明确 Mock 策略

**当前问题**：
"渐进式真实度"部分提到 Mock，但没有明确说明 Mock 只用于外部不可控依赖。

**修改**：
在"核心测试理念"的"渐进式真实度"部分，增加强调：

```markdown
### 渐进式真实度

当系统涉及**金钱、时间流、外部副作用**时，采用渐进式验证：

```
Mock/本地模拟  →  服务商沙盒环境  →  真实环境最小验证
（最安全）        （接近真实）        （需用户授权）
```

**核心原则**：
- **竭尽全能使用真实环境** — 所有可控的资源（本地 DB、本地服务、内部 API）都使用真实环境
- **Mock 只用于完全不可达的外部依赖** — 如真实支付、真实短信、真实邮件等有副作用且无沙盒的系统
- **优先用服务商提供的沙盒** — 如支付宝沙盒、券商模拟盘，最省力、最接近真实
- **不 Mock 内部逻辑** — 我们不做单元测试式的 Mock，所有内部模块都走真实调用

**什么需要 Mock/沙盒**：
- ✅ 真实支付（有金钱交易）→ 使用支付沙盒或 Mock
- ✅ 真实短信/邮件（有外部发送）→ 使用服务商测试号或 Mock
- ✅ 第三方 OAuth（无测试环境）→ Mock 回调
- ✅ 定时任务（无法等待）→ 手动触发或时间 Mock

**什么不需要 Mock**：
- ❌ 本地数据库 → 使用真实 PostgreSQL/MySQL Docker
- ❌ 本地服务 → 启动真实服务，不 Mock
- ❌ 内部 API → 调用真实 API，不 Mock 响应
- ❌ 内部模块 → 集成测试，不 Mock 依赖
```

**影响文件**：
- `skills/echo-tdd-plan/SKILL.md`

## Phase 3: 用例充实

### 3.4 优先级标注

为**每个用例**标注优先级（P0/P1/P2）：

## 4. 风险和缓解

### 风险 1：禁止 plan 探测可能导致信息不足

**表现**：Phase 3 不能探测，可能收集的信息不准确

**缓解**：
- Phase 3 提问更详细，提供更多选项
- verify 阶段会验证所有信息，如有误差会自动纠正
- 用户可以在 Phase 4 调整决策

### 风险 2：MacOS APP 用户可能不知道 mac-use

**表现**：即使强调了 mac-use，用户可能还是想用其他方案

**缓解**：
- 在项目类型识别中明确推荐
- 在可观测性方案中说明为什么 mac-use 是最佳选择
- 如果用户坚持用其他方案，尊重用户选择

### 风险 3：Mock 策略误解

**表现**：用户可能仍然想 Mock 内部逻辑

**缓解**：
- 在多处强调"不 Mock 内部逻辑"
- 提供清晰的"需要 Mock"和"不需要 Mock"的对比示例
- 在 Phase 4 决策确认时再次强调
