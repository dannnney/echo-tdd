# 计划：构建阶段三 SKILL — 测试用例 + 测试数据方案生成

## Context

阶段一（`/testability`）产出测试策略文档，阶段二（`/testability-scaffold`）验证环境并生成脚手架代码。阶段三的目标是：

**基于前两阶段的实际产出，生成个性化的测试用例文档（纯文本，非代码）+ 测试数据方案。**

关键约束：
- 产出是**测试用例文本**，不是测试代码（代码化属于后续阶段）
- 用例和数据方案必须**适配用户的实际环境和可观测性通道**（从 Phase 1 策略 + Phase 2 探测报告提取）
- 支持**真实数据采样**——用已验证的通道（如 SDK、DB CLI）读取环境中的真实数据结构，用于生成更贴合实际的用例

---

## 用户确认的方向

| 议题 | 选择 |
|------|------|
| 用例覆盖策略 | 混合方案：规格推导为骨架 + 用户旅程补故事 + 风险标优先级 + 维度矩阵查漏 + 真实数据采样 |
| 数据组织 | 分层策略：不同测试层用不同数据策略（只读层共享、写操作层独立、端到端完整生命周期） |
| 数据规模 | 尽可能丰富完善 + 分级执行：核心用例优先，边缘场景放后面验收 |
| 缺失项 | 执行顺序与依赖（基于环境个性化）；不在此阶段解决代码转化 |

## SKILL 设计

### 文件结构

```
skills/testability-cases/
├── SKILL.md                    ← 主 skill prompt
├── coverage-patterns.md        ← 用例覆盖策略模式库（如何系统化推导用例）
├── data-blueprint-guide.md     ← 测试数据蓝图设计指导
├── output-template.md          ← 测试用例文档输出模板
└── examples/
    └── cli-feishu-cases.md     ← fz 项目的完整阶段三示例
```

### 输入

- **必需**：阶段一策略文档（如 `@docs/test-strategy.md`）
- **必需**：阶段二探测报告 / 脚手架代码目录（如 `@test/`）
- **可选**：用户补充的业务场景或优先级说明

### 核心原则

```
策略驱动，用例基于阶段一的分层和可观测性方案
环境适配，数据准备/执行/验收方式适配阶段二确认的实际通道
真实采样，有条件时从环境中采样真实数据结构辅助用例设计
丰富完善，用例和数据尽可能全面覆盖，不刻意精简
分级执行，核心场景优先验收，边缘场景排后
分层策略，不同测试层用不同的数据生命周期
文本产出，只生成用例文档不生成代码
```

### 流程

```
Phase 0: 输入解析
    │  ← 读取策略文档 + 探测报告，提取关键约束
    ▼
Phase 1: 环境采样（可选，如果通道允许）
    │  ← 通过已验证的通道采样真实数据结构和值分布
    │  ← 用于辅助后续的用例设计
    ▼
Phase 2: 用例骨架生成（规格推导）
    │  ← 对每个被测功能，系统化推导输入空间
    │  ← 等价类划分 + 边界值 + 属性建模
    ▼
Phase 3: 用例充实（多维度补全）
    │  ← 用户旅程补充端到端故事
    │  ← 真实采样数据补充现实场景
    │  ← 维度矩阵交叉查漏
    │  ← 风险标注优先级（P0/P1/P2）
    ▼
Phase 4: 数据蓝图生成
    │  ← 基于所有用例汇总数据需求
    │  ← 按测试层分配数据生命周期
    │  ← 生成数据蓝图文档
    ▼
Phase 5: 用户审阅
    │  ← 展示用例清单和数据蓝图
    │  ← 用户确认/修正/补充
    ▼
输出：测试用例文档 + 数据蓝图
```

---

### Phase 0: 输入解析

从前两阶段的产出中提取：

| 来源 | 提取什么 | 用途 |
|------|---------|------|
| 策略文档 第 1 节 | 测试范围和排除项 | 确定用例边界 |
| 策略文档 第 3 节 | 测试分层 + 可观测性方案（触发×观测组合） | **决定每个用例的触发方式和验收方式** |
| 策略文档 第 4 节 | 数据流闭环（准备→送入→验证→清理） | 决定数据蓝图的结构 |
| 策略文档 第 5 节 | 认证方案 | 认证相关用例 + 数据准备的认证前提 |
| 探测报告 | 各通道的 PASS/FAIL/WARN | **决定哪些通道实际可用，用例必须适配** |
| 脚手架代码 | data-factory、client 的接口 | 数据蓝图中引用已有的工厂函数 |

**关键**：如果探测报告中某个通道 FAIL，相关用例的验收方式必须降级到其他可用通道。这是"个性化"的核心。

### Phase 1: 环境采样（可选）

当阶段二已验证某些通道可用时，利用这些通道采样真实环境中的数据：

**采样目标**：
- 数据结构（如飞书网盘中实际有多深的嵌套、实际文件名包含哪些字符）
- 值分布（如文档类型的占比：docx 80%、sheet 15%、mindnote 5%）
- 边界案例（如最长的文件名、最深的嵌套层级）

**采样方式**：
- 通过 SDK/API 查询现有数据结构
- 通过 DB CLI 查询数据分布（如 `SELECT type, COUNT(*) FROM files GROUP BY type`）
- 通过 UI 截图或浏览器 snapshot 获取界面数据

**采样输出**：
```
环境数据采样报告：
- 网盘最大嵌套深度：4 层
- 文件名最大长度：47 字符
- 文档类型分布：docx(73%), sheet(18%), bitable(6%), mindnote(3%)
- 单个文件夹最多子项：127 个
- 存在中文文件名：是（占比 65%）
- 存在特殊字符文件名：是（空格、括号）
```

这些数据用于 Phase 2/3 中生成更真实的用例。

**注意**：采样是只读操作，不创建/修改任何数据。如果通道不可用或用户拒绝，跳过此步骤，用通用假设。

### Phase 2: 用例骨架生成

对策略文档中每个被测功能，系统化推导用例：

**步骤**：
1. 识别输入参数和变量维度
2. 对每个维度做等价类划分
3. 提取边界值
4. 用属性建模描述数据特征
5. 裁剪笛卡尔积（pairwise 或风险驱动裁剪）

**属性建模示例**：
```
fz ls <path> 的属性模型：

path 属性：
  - 路径域: /drive/ | /wiki/
  - 路径深度: 0(根) | 1 | 2+ (深层嵌套)
  - 路径字符集: ASCII | 中文 | 特殊字符 | 混合
  - 路径存在性: 存在 | 不存在
  - 末尾斜杠: 有 | 无

目标目录属性：
  - 内容数量: 0(空) | 少量(1-5) | 临界(分页阈值) | 大量(超分页)
  - 内容类型: 纯文件夹 | 纯文档 | 混合
  - 文档子类型: docx | sheet | bitable | mindnote
```

**每个功能产出**：一张用例表，包含正常路径、异常路径、边界值用例。

### Phase 3: 用例充实

在骨架基础上，从三个方向补充：

**3.1 用户旅程**
设计 2-3 个完整的端到端故事，覆盖真实使用场景。

**3.2 真实数据补充**
如果 Phase 1 采样了真实数据：
- 将采样发现的边界值（如最长文件名 47 字符）加入用例
- 将采样发现的类型分布用于设计更真实的测试数据组合
- 将采样发现的实际嵌套深度作为深层路径测试的参考

**3.3 维度矩阵查漏**
将所有维度列成矩阵，检查是否有未覆盖的组合。

**3.4 风险标注**
为每个用例标注优先级：
```
P0（冒烟）— 核心正常路径，每次提交必须通过
P1（回归）— 重要异常路径 + 边界值，版本发布前跑
P2（完整）— 罕见边界 + 兼容性，定期跑
```

### Phase 4: 数据蓝图生成

汇总所有用例的数据需求，按测试层组织：

**数据蓝图结构**：
```
## 数据蓝图

### 全局数据（所有层共享）
- 认证凭证（.env.test 中的 APP_ID/APP_SECRET）

### 第 N 层数据
| 数据名称 | 类型 | 详情 | 创建方式 | 生命周期 | 被引用用例 | 优先级 |
|---------|------|------|---------|---------|-----------|-------|
| test_root | 文件夹 | 测试根目录 | SDK createFolder | beforeAll / afterAll | L2.LS.*, L3.MKDIR.* | P0 |
| three_folders | 子文件夹×3 | alpha/beta/gamma | SDK ×3 | 随 test_root | L2.LS.02 | P0 |
| many_folders | 子文件夹×51 | f001...f051（分页测试） | SDK ×51 | 随 test_root | L2.LS.08 | P1 |
| unicode_folders | 子文件夹×5 | 中文/日文/emoji/空格/括号 | SDK ×5 | 随 test_root | L2.LS.06-07 | P1 |
| deep_nesting | 4层嵌套 | a/b/c/d | SDK 递归创建 | 随 test_root | L4.PATH.03 | P1 |
| extreme_name | 文件夹 | 300字符名称 | SDK | 随 test_root | LX.ERR.04 | P2 |

### 分层数据生命周期
- 第 1 层（认证）：无需数据
- 第 2 层（只读）：beforeAll 创建全量 fixture → 所有只读用例复用 → afterAll 清理
- 第 3 层（写操作）：beforeEach 创建父目录 → 用例执行 → afterEach 清理
- 第 4 层（端到端）：每个用例独立管理完整生命周期

### 数据量统计
按优先级汇总：
- P0 数据：~20 项（核心验证最少数据集）
- P1 数据：~50 项（覆盖边界值和重要变体）
- P2 数据：~30 项（罕见边界和兼容性）
- 总计：~100 项
```

### Phase 5: 用户审阅

展示用例清单和数据蓝图，征求用户确认：
- 覆盖度是否足够？
- 优先级标注是否合理？
- 数据蓝图是否可行？
- 有无遗漏的业务场景？

## `coverage-patterns.md` 设计要点

用例覆盖策略的模式库，提供以下工具：

### 1. 等价类划分模板
如何对输入参数做有效等价类和无效等价类划分。

### 2. 边界值分析模板
如何识别边界：空/非空、0/1/max、分页阈值、字符集边界。

### 3. 属性建模框架
如何对被测功能建立属性模型，列出所有可变维度。

### 4. 笛卡尔积裁剪策略
- Pairwise（两两覆盖）
- 风险加权裁剪（高风险组合保留、低风险裁剪）

### 5. 环境采样集成
如何利用真实数据采样结果增强用例：
- 采样值直接作为测试输入
- 采样分布指导用例权重
- 采样边界补充边界值分析

### 6. 用户旅程设计模板
如何从角色 → 目标 → 步骤 → 期望 设计端到端故事。

## `data-blueprint-guide.md` 设计要点

### 1. 分层数据策略
不同测试层的数据生命周期模式。

### 2. 数据隔离原则
命名前缀、命名空间、独立环境。

### 3. 数据需求汇总规范
如何从用例中提取数据需求并汇总为蓝图。

### 4. 数据与可观测性通道的适配
- 有 DB 访问：数据直接 SQL 插入
- 有 SDK：数据通过 SDK 创建
- 只有 API：数据通过 API 创建
- 只有 UI：数据通过浏览器操作创建
- 根据阶段二探测结果选择策略

### 5. 丰富完善 + 分级执行原则
数据不刻意精简，而是尽可能覆盖各种变体：
- 每个等价类至少一条数据
- 每个边界值至少一条数据
- 真实采样发现的变体都要覆盖
- 通过优先级分级控制执行节奏：
  - **P0 数据**：核心正常路径所需，每次都创建
  - **P1 数据**：重要异常和边界值所需，回归时创建
  - **P2 数据**：罕见边界和兼容性所需，完整验收时创建

## `output-template.md` 设计要点

测试用例文档输出模板，包含：

### 1. 文档头信息
生成时间、依赖的策略文档和探测报告版本。

### 2. 通用约定
触发通道、观测通道、数据隔离前缀、清理策略——**从策略文档第 3/4 节提取，不是硬编码**。

### 3. 环境采样摘要（如有）
真实数据的结构和分布发现。

### 4. 分层用例表
每层包含：
- 前提条件
- 数据准备方式
- 用例表（编号、名称、前置条件、触发、预期结果、观测方式、优先级）

### 5. 用户旅程
2-3 个端到端故事。

### 6. 维度覆盖矩阵
维度交叉表，标注哪些组合已覆盖、哪些未覆盖及原因。

### 7. 数据蓝图
分层的数据需求清单 + 生命周期。

### 8. 用例统计
按层、按命令、按优先级的统计表。

## 验证方式

1. 用 fz 项目的 test-strategy-v3.md + test/ 脚手架作为输入，手动触发 Phase 0-5
2. 检查生成的用例是否覆盖了 test-cases-v1.md 中的所有场景

5. 如有真实飞书环境，测试 Phase 1 采样流程
