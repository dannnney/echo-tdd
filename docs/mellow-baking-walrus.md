# Testability Skill 设计文档：自反馈测试方案生成器（阶段一）

## Context

在 Vibe Coding 工作流中，AI 生成的代码需要一套自反馈机制来发现问题并持续修复。这个机制的核心前提是：**必须有一套可执行的测试方案**。

不同开发者的环境千差万别——有人全栈本地开发，有人只做后端 API，有人的数据库在远程且只读，有人的微服务本地根本跑不起来。因此，测试方案不能是模板化的，必须根据每个开发者的实际环境动态生成。

本设计覆盖**阶段一：需求理解 → 测试方案生成**。后续阶段二（环境探测验证）和阶段三（测试用例生成 + 数据准备）将在单独的设计中处理。

---

## Skill 概述

- **名称**: `testability`
- **触发方式**: `/testability [可选参数]`
- **输入**: 自然语言需求描述 / spec 文档路径 / 无输入（纯代码库推断）
- **输出**: 一份 markdown 测试方案文档
- **实现形式**: Skill prompt（SKILL.md）+ 辅助参考文件 + 运行时可动态生成探测脚本

---

## 整体流程

```
Phase 0: 输入解析
    │  ← 解析用户传入的参数（需求描述/文档路径/无输入）
    ▼
Phase 1: 代码库扫描（智能推断）
    │  ← Agent 自动扫描项目结构，推断六大维度的信息
    ▼
Phase 2: 需求理解与澄清
    │  ← 结合代码库信息 + 用户输入，展示推断结果
    │  ← 用户确认/修正推断
    │  ← 讨论和澄清需求细节
    ▼
Phase 3: 维度补全（逐维度提问）
    │  ← 对仍然缺失的维度逐一提问
    │  ← 可动态生成探测脚本验证（如测试 DB 连通性）
    ▼
Phase 4: 方案生成 + 用户审阅
    │  ← 生成测试方案 markdown 文档
    │  ← 用户审阅确认
    ▼
输出：测试方案.md
```

## 六大核心维度（组合化设计）

每个维度有多种可能的值，真实场景是这些维度值的组合。Skill 的目标是确定每个维度的实际值。

### 1. 需求维度
用户要开发/测试什么功能。信息来源：
- 用户直接描述
- 用户提供的 spec 文档
- 从代码库最近的改动推断

### 2. 代码仓库定位
- **当前 repo 类型**：前端 / 后端 / 全栈 / 桌面 / 微服务中的某个服务 / 移动端
- **关联 repo**：是否有其他 repo 可以联动（如后端开发者有前端 repo 访问权限）
- **代码权限**：读写 / 只读 / 可部署

### 3. 基础设施维度

**服务端：**
| 可能值 | 说明 |
|--------|------|
| 本地可运行 | 服务可以在本地启动 |
| 本地跑不起来 | 微服务依赖太多或环境太复杂 |
| 远程 dev 环境可用 | 有可访问的开发/测试环境 |
| 只有生产环境 | 无法用于测试 |

**数据库：**
| 可能值 | 说明 |
|--------|------|
| 本地 + 可清空 | 完全控制，可以重置 |
| 本地 + 只读 | 本地有但不能随意修改 |
| 远程 + 可读写 | 远程数据库有写权限 |
| 远程 + 只读 | 只能查询验证 |
| 无访问权限 | 完全不能直接访问数据库 |

**前端：**
| 可能值 | 说明 |
|--------|------|
| 本地开发可运行 | 前端在本地跑 |
| 本地部署 + 连远程 API | 前端本地，API 在远程 |
| 远程 dev 环境 | 前端部署在远程 |
| 无前端 | 纯 API / CLI 项目 |

### 4. 验证通道维度
哪些验证手段可用：
- **浏览器**（Playwright/Puppeteer）— 可访问的 URL 是什么
- **数据库 CLI**（psql/mysql/mongo）— 连接方式
- **API**（curl/fetch）— 端点地址
- **WebSocket** — 服务器地址和事件
- **桌面自动化**（mac-use 等）— 适用于桌面应用
- **移动端**（ADB 等）— 适用于移动应用
- **日志/控制台** — 可观测性

### 5. 认证维度
| 可能值 | 说明 |
|--------|------|
| 可以自注册 | Skill 可以自己注册账号获取登录态 |
| 提供用户名密码 | 用户提供凭据 |
| Cookie/Token 导入 | 用户提供认证凭据文件 |
| Auth 文件导入 | 如 `.auth` 文件 |
| 不需要认证 | 公开服务或本地开发模式 |
| OAuth/SSO | 需要特殊处理的第三方认证 |

### 6. 数据策略维度
| 可能值 | 说明 |
|--------|------|
| 可以清空重建 | 完全控制，可以 drop + seed |
| 只能追加 | 可以插入测试数据但不能删除 |
| 只能用现有数据 | 不能修改任何数据 |
| 需要 seed 脚本 | 有现成的数据初始化脚本 |
| 需要通过 UI 创建 | 测试数据需要通过页面操作创建 |

## Phase 详细设计

### Phase 0: 输入解析

解析用户在触发 `/testability` 时传入的参数：
- 如果是文档路径（如 `@docs/spec.md`）→ 读取文档
- 如果是自然语言描述 → 记录需求
- 如果无参数 → 准备纯代码库推断模式

### Phase 1: 代码库扫描

指导 Agent 扫描当前代码库，推断六大维度的信息。Agent 自行决定扫描策略，但需覆盖以下推断目标：
- 项目类型和技术栈
- 数据库类型和配置方式
- 已有的测试基础设施
- API/路由结构
- 认证方式
- 部署和运行方式

扫描完成后，Agent 将推断结果整理为结构化的「环境画像」。

### Phase 2: 需求理解与澄清

1. 展示推断结果给用户，格式清晰
2. 让用户确认/修正每项推断
3. 结合需求描述，讨论测试目标：
   - 哪些功能需要测试？
   - 测试的深度和广度？
   - 有没有特别关注的风险点？

### Phase 3: 维度补全

对仍然缺失的维度逐一提问。原则：
- **能推断的不问** — 已从代码库或用户确认的不再重复
- **一次只问一个维度** — 不要一次抛出一堆问题
- **提供选项** — 尽可能给选择题
- **可动态探测** — 对某些维度可以通过运行脚本来验证（如测试数据库连通性）

### Phase 4: 方案生成

基于完整的维度信息，生成测试方案 markdown。方案内容包含：

1. **需求概述** — 测试目标和范围
2. **环境画像**
   - 代码仓库定位（repo 类型、关联 repo、权限）
   - 技术栈（语言/框架/数据库/测试工具）
   - 基础设施（服务端、数据库、前端的具体情况）
3. **验证通道** — 每个可用通道的类型、工具、连接方式
4. **认证方案** — 具体的认证获取步骤
5. **测试边界** — 需要验证的层 vs 不需要的层
6. **数据策略** — 数据准备和清理方式
7. **测试目标清单** — 具体的端点/页面/功能列表
8. **环境前置条件** — 阶段二需要验证的条件 checklist

生成后交由用户审阅确认。

## SKILL.md 核心原则

1. **推断优先** — 能从代码库推断的绝不问用户
2. **一次一问** — 逐维度提问，不要一次性抛出所有问题
3. **组合化思维** — 不要用角色标签（如"大厂开发者"），用维度组合描述实际情况
4. **不问隐私** — 不问团队规模等无关信息
5. **动态探测** — 对可验证的维度，优先用脚本探测而非提问
6. **输出可执行** — 方案中的工具和命令必须具体到可直接使用

## 实现计划

### Step 1: 创建 SKILL.md
主 skill prompt 文件，包含：
- Frontmatter（name: testability, description）
- 核心原则
- Phase 0-4 流程定义（含 Graphviz 流程图）
- 引用 dimensions.md 和 output-template.md
- 动态探测脚本生成指导

### Step 2: 创建 dimensions.md
六个维度的详细定义文件：
- 每个维度的所有可能值
- 推断线索（从什么文件/配置可以推断）
- 提问模板（当需要问用户时如何问）
- 组合约束（某些值的组合是否矛盾）

### Step 3: 创建 output-template.md
测试方案输出的 markdown 模板

### Step 4: 创建 examples/ 目录
3-5 个典型维度组合的示例，供 Agent 参考

---

## 验证方式

1. **Skill 语法验证**：确保 SKILL.md 符合 superpowers skill 的 frontmatter 和结构规范
2. **流程测试**：用一个真实项目（如 Next.js 全栈项目）触发 `/testability`，检查：
   - Phase 1 能否正确扫描和推断
   - Phase 2 展示的推断结果是否合理
   - Phase 3 的提问是否精准（不问已知信息）
   - Phase 4 生成的方案是否包含所有维度
3. **边界测试**：用空项目、纯前端项目、纯 API 项目等不同场景测试
4. **输出质量**：检查生成的测试方案是否具体到可直接执行
