# 用例覆盖策略模式库

本文档提供系统化生成测试用例的方法和模式。Agent 按照 SKILL.md 中的流程，在 Phase 2（骨架生成）和 Phase 3（充实）中使用这些模式。

---

## 1. 等价类划分

将每个输入参数的取值空间划分为若干等价类，每个类中选一个代表值作为测试输入。

### 有效等价类

功能应该正常处理的输入：

```
参数：文件路径
有效等价类：
  - 根路径（/drive/）
  - 一层路径（/drive/folder1/）
  - 多层嵌套路径（/drive/a/b/c/）
  - 中文路径（/drive/项目文档/）
  - 含空格路径（/drive/my folder/）
```

### 无效等价类

功能应该报错或优雅处理的输入：

```
参数：文件路径
无效等价类：
  - 空字符串
  - 不存在的路径
  - 无效前缀（/invalid/）
  - 超长路径（超过系统限制）
  - 非法字符（如果有的话）
```

### 原则

- 每个等价类至少一个用例
- 有效等价类可以组合；无效等价类一次只测一个（避免多个错误互相掩盖）
- 优先从真实采样中发现等价类的实际分布

---

## 2. 边界值分析

在每个等价类的边界处设置测试点。

### 常见边界

| 维度 | 边界点 |
|------|--------|
| 数量 | 0、1、阈值-1、阈值、阈值+1、MAX |
| 长度 | 0（空）、1、MAX-1、MAX、MAX+1 |
| 深度 | 0（根）、1、常见最大深度、超过最大 |
| 分页 | 恰好一页、一页+1、多页 |
| 字符 | ASCII 最小（空格）、ASCII 最大（~）、Unicode 起点、emoji |

### 结合采样数据

如果 Phase 1 采样了真实数据：
- 用**实际最大值**替代理论最大值（如采样发现最长文件名 47 字符，就测 47、48）
- 用**实际分页阈值**（如 API 返回 page_size=50，就测 49、50、51）
- 采样发现的**异常边界**直接加入（如发现有 0 字节文件、空标题文档）

## 3. 属性建模框架

对每个被测功能，建立属性模型——列出所有可变维度和可能的值。

### 建模步骤

```
1. 列出功能的所有输入参数
2. 对每个参数，识别其属性维度
3. 对每个维度，列出所有可能的值
4. 标注维度之间的约束关系（如 /wiki/ 路径必须包含空间名）
5. 标注哪些维度是正交的（可以自由组合）、哪些是有依赖的
```

### 模型格式

```
<功能名> 属性模型：

【输入参数】
参数 1：<名称>
  - 维度 A: 值1 | 值2 | 值3
  - 维度 B: 值1 | 值2
  约束：维度 A = 值1 时，维度 B 只能是 值1

参数 2：<名称>
  - 维度 C: 值1 | 值2 | 值3 | 值4

【环境变量】（非参数，但影响行为）
  - 认证状态: 有效 | 过期 | 无凭证
  - 网络状态: 正常 | 超时 | 断连
  - 目标资源状态: 存在 | 不存在 | 无权限

【输出】
  - 正常输出维度: [stdout 格式、返回值类型]
  - 错误输出维度: [stderr 格式、exit code]
```

## 4. 笛卡尔积裁剪

属性建模后，理论上的全组合可能很大。需要智能裁剪。

### Pairwise 覆盖（两两覆盖）

确保任意两个维度的所有值对都至少在一个用例中出现。

```
示例：3 个维度，每个维度 3 个值
全组合：3×3×3 = 27 个用例
Pairwise：~9-12 个用例（覆盖所有两两组合）
```

适用于维度多但维度间交互弱的场景。

### 风险加权裁剪

对每个维度组合评估风险，高风险保留、低风险裁剪：

```
风险评估：
  - 正常值 × 正常值 → 低风险（可裁剪，保留一个代表）
  - 边界值 × 正常值 → 中风险（保留）
  - 边界值 × 边界值 → 高风险（必须保留）
  - 无效值 × 任何值 → 高风险（必须保留）
```

### 约束排除

排除不可能出现的组合：

```
排除：
  - 路径域 = /wiki/ AND 文档类型 = file（Wiki 没有 file 类型）
  - 认证状态 = 无凭证 AND 操作 = 写操作（不可能成功）
```

## 5. 环境采样集成

如何将 Phase 1 的采样结果融入用例设计。

### 采样值直接作为测试输入

```
采样发现：文件名中存在 "Q3 报告（修订版）.docx"
→ 新增用例：路径中包含中文+英文+括号+空格的混合名称
→ 测试输入直接使用类似的字符组合
```

### 采样分布指导用例权重

```
采样发现：docx 占 73%, sheet 占 18%, bitable 占 6%, mindnote 占 3%
→ docx 用例标 P0（最常见）
→ sheet 用例标 P1
→ bitable 和 mindnote 用例标 P1-P2
→ 不要忽略低占比类型——它们往往是 bug 多发区
```

### 采样边界补充边界值

```
采样发现：最深嵌套 4 层，最多子项 127 个
→ 深层嵌套测试用 4 层作为参考（而非通用的 3 层假设）
→ 分页测试考虑 127 这个数字（接近什么分页阈值？）
```

### 采样异常发现

```
采样发现：有些文件夹名为空字符串、有些文档标题含换行符
→ 这些是真实存在的边界情况
→ 直接加入 P1 用例
```

## 6. 用户旅程设计

从真实用户角色和目标出发，设计完整的使用故事。

### 旅程格式

```
### 旅程：[旅程名称]

**角色**：[谁在用这个功能]
**目标**：[他要完成什么任务]
**前置条件**：[起始状态]

**步骤**：
1. [操作] → [期望结果]
   映射用例：[L2.LS.01]
2. [操作] → [期望结果]
   映射用例：[L2.CAT.01]
...

**终态**：[完成后的状态]
**覆盖的层**：[第 2 层 + 第 3 层]
**优先级**：P0（核心使用场景）
```

### 旅程设计原则

- 从**真实角色**出发（不是"测试人员"，而是"新员工"、"项目经理"、"运维"）
- 每个旅程覆盖**跨层操作**（如先 ls 查看，再 cat 阅读，再 mkdir 创建）
- 旅程中每一步都**映射到用例表**中的具体用例
- 如果旅程中有步骤没有对应用例 → 补充新用例
- 2-3 个旅程足够，不要过多

## 7. 维度覆盖矩阵

将所有关键维度列成交叉表，检查覆盖度。

### 矩阵格式

```
| 用例编号 | 路径域 | 路径深度 | 字符集 | 内容量 | 内容类型 | 操作 | 认证 |
|---------|--------|---------|--------|--------|---------|------|------|
| L2.LS.01 | /drive/ | 根 | ASCII | 正常 | 混合 | ls | 有效 |
| L2.LS.09 | /wiki/ | 根 | 中文 | 正常 | 混合 | ls | 有效 |
| L3.MKDIR.02 | /drive/ | 1层 | 中文 | — | — | mkdir | 有效 |
| ... |
```

### 查漏方法

1. 对每对维度，检查是否所有值对都出现过
2. 标注未覆盖的组合
3. 对每个未覆盖组合判断：
   - **有价值** → 补充用例
   - **不可能组合** → 标注"约束排除"
   - **低价值** → 标注"已知未覆盖，风险可接受"
